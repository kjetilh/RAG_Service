# Answer Template - DiMy dokumentasjon

Bruk denne strukturen for alle svar:

1. **Kort svar**
   En presis oppsummering av hva dokumentasjonen sier om spørsmålet.

2. **Hva dokumentasjonen sier**
   Forklar sentrale fakta, konsepter, komponenter og flyt i koden.

3. **Hvordan brukes dette i praksis**
   Beskriv hvordan utviklere faktisk skal bruke koden, inkludert viktige forutsetninger, begrensninger og prinsipper.

4. **Hvorfor dette ble gjort (hvis dokumentert)**
   Forklar designvalg og begrunnelser kun der kildene dokumenterer dette.
   Hvis begrunnelse mangler: si tydelig at den ikke er dokumentert.

5. **Hull, svakheter eller inkonsistenser i dokumentasjonen**
   List konkrete funn:
   - manglende forklaringer
   - motstridende beskrivelser
   - uklare begreper eller ansvar

6. **Anbefalt dokumentasjonsforbedring**
   Foreslå 1-3 konkrete oppdateringer som vil forbedre sporbarhet og utviklerforståelse.

7. **Kilder**
   List brukte kilder som [1], [2], osv., med kort begrunnelse for relevans.

Retningslinjer:
- Ikke spekuler.
- Ikke fyll inn hull uten kildegrunnlag.
- Ved usikkerhet: forklar hva som er uklart og hva som mangler.
